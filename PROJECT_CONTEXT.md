# The Needle Project — Context & Operations Guide

This document is the single source of truth for how this project is built, deployed, and maintained. Read it first before doing any work in this folder. It is written so a fresh assistant session can act without re-deriving anything.

Last updated: 2026-07-18 (through Update 23).

---

## 1. What this project is

**The Needle Project** is an independent, live-updating tracker for the 2026 U.S. elections. Each tracked state gets a probability "needle" (win probability) that runs on polling now and switches to real county returns on election night.

- **Live site:** https://theneedleproject.vercel.app
- **GitHub repo:** `cblais75/maine-needle` (this is the **source of truth** — see §3)
- **Hosting:** Vercel, which **auto-deploys the `main` branch on every push**. There is no manual deploy button; pushing to GitHub is the deploy.
- **Owner:** Colin (writer, Food & Politics on Substack: https://foodandpolitics.substack.com). Not a coder. Core value: **data and facts over bias.** If the model favors a Republican, the needle shows that honestly. Never tune numbers to a preferred outcome.

---

## 2. The one rule that matters most

**The app reads `public/current.json`, NOT `data/polls.json`.**

`data/polls.json` is raw input. It is compiled into `public/current.json` by the script `npm run polls`. **Editing `polls.json` changes nothing on the site until `npm run polls` runs and the regenerated `current.json` is committed.**

This has bitten the project more than once (polls added, nothing changed on the site). Whenever you touch `polls.json`, or deploy any code package, **run `npm run polls` before building and committing.**

## 4. Standard commands

Run these from the repo root:

- `npm install` — only needed when dependencies change (rare; e.g., adding analytics).
- `npm run polls` — recompile `public/current.json` from `data/polls.json`. **Run after any `polls.json` edit and after copying any code package.**
- `npm run build` — verify the project builds. A clean build means it *compiles*; it does **not** guarantee it *runs* (see §9).
- `git add -A && git commit -m "..."` then run **`git-push-final.bat`** to push. Pushing triggers the Vercel deploy.

Keep commit messages short and neutral: `Update N`, `Polls`, `Briefing`. (Charged political text in commits has tripped an automated content filter before.)

## 5. Deploying a code update (from a packaged zip)

Claude in chat produces update packages (zips named `the-needle-project_DATE_update-N_description.zip`). To deploy one:

1. Copy the package's files into the project, **but do NOT overwrite anything in `data/`** (that folder holds Colin's real polls and baselines). If the update note says a *new* file under `data/` is included (e.g. a new baseline), copy that specific new file only.
2. If the package adds a dependency (the note will say so), run `npm install`.
3. Run `npm run polls` (regenerates `current.json` from the real `polls.json`).
4. Run `npm run build` to confirm it builds.
5. `git add -A`, commit `Update N`, run `git-push-final.bat`.
6. After Vercel finishes, **load the live site once** to confirm it renders (not just that it built).

**Why the "don't overwrite data/" rule exists:** packages are built in a separate environment. In the past, copying a package's `polls.json`/`current.json` clobbered Colin's real poll data, and once wiped the analytics change. Packages are now built from a fresh repo clone so they carry the deployed state, but the safe habit stands: never let a package overwrite `data/`, and always re-run `npm run polls`.

## 6. Posting a Briefing (Colin's written analysis)

The **Briefing** tab is Colin's editorial voice — written commentary that lives alongside the math. Posts are stored in `public/briefing.json` as an array, newest shown first.

To add a post, append an object to the array (keep all existing entries):

```json
{
  "date": "YYYY-MM-DD",
  "title": "Post title",
  "body": "Paragraph one.\n\nParagraph two. Blank lines (\\n\\n) become paragraph breaks.",
  "links": [
    {"url": "https://foodandpolitics.substack.com/p/slug", "label": "Read the full piece on Food & Politics"},
    {"url": "https://foodandpolitics.substack.com", "label": "Subscribe to Food & Politics"}
  ]
}
```

`links` is optional; each renders as a clickable button at the bottom of the card. Then `npm run build`, commit `Briefing`, push. (No `npm run polls` needed for a Briefing-only change, but it's harmless to run.)

Typical pattern: Colin publishes on Substack, and a condensed version is posted here in his voice with links back to the full article.

| `cd1` | Maine House District 1 | Pingree (D) vs Russell (R) | |

| `nc_sen` | North Carolina Senate | Cooper (D) vs Whatley (R) | |
| `oh_sen` | Ohio Senate | Brown (D) vs Husted (R) | |
| `tx_sen` | Texas Senate | Talarico (D) vs Paxton (R) | |
| `ia_sen` | Iowa Senate | Turek (D) vs Hinson (R) | |
| `ga_sen` | Georgia Senate | Ossoff (D) vs Collins (R) | Dec 1 runoff if no one tops 50% |
| `ne_sen` | Nebraska Senate | Osborn (**I**) vs Ricketts (R) | **Enter Osborn as `dem`, Ricketts as `rep`** |
| `ak_sen` | Alaska Senate | Peltola (D) vs Sullivan (R) | RCV; polling tracked but no needle |

### Poll entry format
Each poll is one object in its group:
```json
{"pollster": "NYT/Siena", "rating": 0.95, "date": "2026-06-27", "dem": 50, "rep": 43}
```
- `dem`/`rep` are percentages **by party, not by who's leading.** (So Osborn is `dem`, Paxton is `rep`, etc.)
- Governor entries also include `"ind": 8` for Bennett.
- Don't duplicate polls already in the file.

### Quality ratings (0–1)
- **0.9–0.95** — gold standard, nonpartisan: NYT/Siena, Marist, Fox News (Beacon/Shaw — a bipartisan pair, high quality despite the network's brand), Marquette, AP-NORC, ABC/Washington Post.
- **0.8** — solid university/media: Suffolk, Emerson, Quinnipiac, Monmouth, UNH, UMass Lowell.
- **0.5–0.6** — partisan or campaign-affiliated: Quantus, Trafalgar, On Message (R-leaning); PPP, GBAO, Data for Progress (D-leaning); any internal/sponsored poll.
- **0.4** — unknown or unproven shops.

### The refresh procedure
1. Search the web for new public polls (last few weeks) for each live group above.
2. Add new entries to the matching groups. **Skip `senate`.**
3. Run `npm run polls`, report the new averages for any group that changed.
4. If anything was added: commit `Polls`, push with `git-push-final.bat`.

### 7.1 Why Maine Senate (`senate`) is paused
Graham Platner won the primary but **withdrew on July 10, 2026** after sexual assault allegations (which he denies). The Maine Democratic Party is selecting a replacement nominee, to be certified **by July 27**. Until then the Maine Senate needle is **suspended** (there is no real matchup to model), and no polls should be added to `senate`. Reviving it is a code update, not a poll edit (see §10).

## 8. Building a real county baseline

`tx-`, `ia-`, `ga-`, and `ne-` baselines are placeholders (the needle runs statewide until real county data is added). To build a real one, get a CSV of that state's **2024 presidential results by county**, then:

```
node scripts/build-baseline.mjs --out data/XX-baseline.json --file <path-to-county-csv>
```
Expected county counts: Texas 254, Iowa 99, Georgia 159, Nebraska 93. Commit the new baseline (`Update N` or `Baseline`), push. No app code change needed — the race auto-detects the county map and switches from statewide to county-level.

## 9. Hard-won lessons (read these to avoid repeating mistakes)

1. **`current.json` is king.** Editing `polls.json` without `npm run polls` = no visible change. This is the #1 recurring issue.
2. **Never overwrite `data/`** when deploying a package. It holds the real polls and baselines.
3. **A clean `npm run build` does NOT mean the site works.** The build only checks compilation. A runtime error (e.g., a component crashing on render) white-screens the whole site while the build still passes. **Always load the live page after deploying.** (This is why the Senate Control and Wire code carefully skip race types that would crash them.)
4. **Don't write literal `\uXXXX` escapes into JSX strings** — they render as raw text (the "`\u00b7`" bug). Use the real character.
5. **Keep commits neutral** (`Update N`, `Polls`, `Briefing`). Charged political wording has tripped an automated filter.
6. **Push local machine changes before deploying a chat package.** A package can only include what's in the repo; if the local machine has un-pushed edits (this happened with the analytics code), a deploy can overwrite them. Packages are now built from a repo clone to minimize this.
7. **One crash shouldn't kill the site.** (A future error-boundary is a suggested improvement; see §11.)

## 10. The races and how each type behaves

The nav is: **States ▾ (dropdown, alphabetized) | Dashboard | Senate | Briefing | Wire | Polls | Method.** Picking a single-race state opens its needle directly; **Maine** opens a multi-race overview (it has five races).

### Race types (field `type` in each race object)
- **two-way** (default): standard red/blue needle via `compute()` + `rateOf()`.
- **three**: Maine Governor (Pingree/Charles/Bennett). Uses a three-way bar; poll entries need `ind`.
- **rcv**: Alaska. **No needle** (ranked-choice is tabulated ~2 weeks after election night). Shows an explainer panel; polling is still tracked on the Polls tab.
- **tbd**: a suspended/placeholder race with no needle — currently **Maine Senate** (nominee TBD) and **Michigan** (open seat, nominees set at the Aug 4 primary). Rendered as an informational panel with a status chip.

### House effects (poll-center shifts toward the historically underestimated side)
NC +6 → Whatley, Ohio +4 → Husted, Texas +5 → Paxton, Iowa +5 → Hinson, Georgia +2 → Collins, Nebraska +4 → Ricketts. Maine Senate keeps a Collins shift when revived. These reflect real polling-error history and are documented on the site's **Method** tab.

### Nebraska specifics
Dan Osborn runs as an **independent** (colored **teal**, not blue, to signal that). He occupies the `dem` slot in data terms. The needle launches ~**Ricketts 69% / Osborn 31%** on fundamentals (tied polling shifted 4 pts toward Ricketts for Nebraska's R+20 lean and Osborn's 2024 pattern of polling close then losing). Osborn has not said which party he'd caucus with — relevant to the Senate Control math.

## 11. Site features (so you know what exists)

- **Dashboard** — all races at a glance, grouped by state.
- **Senate** ("Control of the Senate") — seat-count math with a majority bar and 50-seat line. Baseline 45 safe-D / 46 safe-R; the 9 tracked races are the live battlefield. Tiles flip automatically when a needle crosses the **97% "Called"** threshold. Also **interactive**: tap any tile to explore what-if scenarios; a gold "SCENARIO" chip appears and one tap returns to live. Osborn counts for neither party unless he declares a caucus ("OSBORN DECIDES THE SENATE" is a real possible banner).
- **Wire** ("Live Wire") — a plain-language election-night ticker generated purely from the model's arithmetic (no AI-written commentary, so it can't hallucinate). Emits: first results, reporting milestones (25/50/75/90/99%), rating changes, lead flips, calls, plus **county-watch** lines (a county's over/under-performance vs its baseline, and trailing-candidate "path to win" notes). County lines are meaningful only on real returns; in the simulation the county numbers are random.
- **Briefing** — Colin's written posts (§6).
- **Polls** — the weighted poll averages per state.
- **Method** — methodology and house-effect explanations.

Simulation: a "Run" control pre-rolls simulated results so the needles, Wire, and Senate tab all move for demos. Real election-night mode reads `api/results.js` + `public/results.json`.

## 15. Election night & stress testing (Update 23)

`api/results.js` is now **multi-state**: one env var per state — `MAINE_RESULTS_URL`, `NC_RESULTS_URL`, `OH_RESULTS_URL`, `TX_RESULTS_URL`, `IA_RESULTS_URL`, `GA_RESULTS_URL`, `NE_RESULTS_URL` (set in the Vercel dashboard). It fetches each in parallel (xlsx or CSV auto-detected), parses with the shared aggregator, and merges. A failing state is skipped and listed in `errors` without taking down the rest. CDN cache is 90s. `TEST_RESULTS_URL` overrides everything and serves a hosted results.json verbatim (for hosted stress tests).

**Stress-test harness** — `scripts/test-night.mjs` fabricates a synthetic night for the six county-mapped states through the *real* parser/aggregator and writes `public/results.json`:
- `node scripts/test-night.mjs --auto --minutes 30` — full night, 0→100%, new snapshot every 20s. Run `npm run dev` alongside and watch the needles, Wire, county-watch, and Senate tab move on "live" data.
- `node scripts/test-night.mjs --frac 0.4` — one snapshot at 40% in. `--seed N` reproduces a run.
- `node scripts/test-night.mjs --reset` — **always run when done**; restores the dormant file. Never commit a test snapshot in `public/results.json`.
